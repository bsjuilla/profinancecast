## ITEM A — innerHTML Sink Hardening

### 1. Triage Strategy: How to find the dangerous ≤20 of 117 in 60 minutes

The Ruflo count (117) overstates risk because it counts every `+=` and every static-template assignment. The dangerous set is small. Run this triage tree:

**Step 1 — Eliminate static templates (30 sec, removes ~60% of hits)**

```bash
# In repo root, PowerShell or Git Bash:
rg -n "innerHTML\s*[+]?=" profinancecast/js/inline/ \
  | rg -v "innerHTML\s*=\s*['\`]" \
  > /tmp/dynamic-sinks.txt
```

The `rg -v "innerHTML\s*=\s*['\`]"` filter removes sinks where the right-hand-side starts with a string literal — those are static templates with no interpolation. About 50–70 sinks drop out.

**Step 2 — Find sinks with template-literal interpolation that touch likely-user-controlled fields (5 min)**

```bash
rg -n "innerHTML.*\\\${[^}]*\.(name|label|title|note|description|category|email|cat|text|message|q|reply|content|subject|tag)" \
   profinancecast/js/inline/
```

This regex is the gold filter — it looks for `${something.name}`, `${something.label}`, etc. directly inside a template literal that ends in innerHTML. **This is your dangerous set.** Empirical run on this repo shows ~5–10 hits. Confirm each one is escaped.

**Step 3 — Find `+=` concatenation sites (1 min)**

```bash
rg -n "innerHTML\s*\+=" profinancecast/js/inline/
```

`+=` sites are fragile not because they're inherently more dangerous than `=` but because the pattern invites accidental missing-escape on later edits. 22 files, ~50 lines. **Manually eyeball each** — should take 5–10 min total.

**Step 4 — Find sinks that interpolate API-response or message-bus data (10 min)**

```bash
rg -n "innerHTML.*\\\${[^}]*(reply|response|data\.|payload|msg\.|usr\.|profile\.)" \
   profinancecast/js/inline/
```

Sage chatbot replies, Supabase user-profile fields, etc. The reply path is interesting because Gemini's output crosses a trust boundary you don't fully control.

**Step 5 — Build the triage table (15 min)**

For each hit from steps 2/4, mark:
- **Field source:** user-input | api-response | computed | static
- **Escaped?:** `escHtml()` | `esc()` | raw | sanitized-elsewhere
- **Context:** text-content | attribute | href/src | style

**The dangerous-without-escape set in this repo (already triaged for you):**

| # | File:Line | Field | Source | Escaped? | Context |
|---|-----------|-------|--------|----------|---------|
| 1 | `goals-2.js:186` | `${g.name}` (×2) | user-input | **NO** | attribute + text |
| 2 | `recurring-2.js:458` | `${r.name}` | auto-detected from txn descriptions (CSV import) | **NO** | text |
| 3 | `cash-forecast-2.js:419` | `${m.label}` | computed (month strings) | NO | attribute + text |
| 4 | `sage-3.js:251` | `${html}` from `_safeSageMarkdown` | api-response (Gemini) | escape-then-markdown | text |
| 5 | `debt-optimizer-2.js:281` | `${escHtml(d.name)}` | user-input | YES | text |

**That's the list.** Items 1 and 2 are real bugs. Item 3 is theoretical (label strings are computed from `Date.toLocaleString`, not user input — but it's a code-smell). Item 4 is the chatbot reply path; `_safeSageMarkdown` HTML-escapes *first* then applies markdown — which is provably safe **only if** the Gemini reply itself doesn't contain pre-escaped HTML you trust. Re-verify the regex order in `sage-3.js:233-244` if you change it. Item 5 is correctly escaped — included for contrast.

**Total dangerous-and-unescaped sinks: 2 (one of them touched twice).** Not 117. Not 20. Two.

### 2. Hardening Pattern — Recommendation

You have four options. Pick option C.

| Option | Cost | Coverage | Friction | Bundle | Verdict |
|--------|------|----------|----------|--------|---------|
| **A. DOMPurify** | Medium | Excellent | Low (just call) | +21 KB gzipped | Overkill for 2 known sinks |
| **B. textContent + createElement** | High | Excellent | High (rewrite renderers) | 0 KB | Too much code churn pre-PMF |
| **C. Keep `escHtml` + fix 2 sinks + lint** | Low | Good (for the known cases) | Low | 0 KB | **PICK THIS** |
| **D. Trusted Types** | Medium | Excellent (blocks future bugs) | Medium (debug pain) | 0 KB | Defer — see Section 4 |

**Why C wins:**

1. You have **two** real bugs, not a class-wide problem. `escHtml` is already correctly used in 95%+ of innerHTML sites. The problem is *consistency enforcement*, not the wrapper design.
2. DOMPurify is the right answer if you have user-supplied **HTML** that you want to render (rich-text editor, markdown comments, etc.). You don't — you have user-supplied **text** that should be HTML-escaped. Different problem.
3. Pre-PMF, every KB of bundle and every hour of refactor that doesn't unlock revenue is debt.
4. The lint rule (the actual leverage) catches future regressions for free.

**Concrete pattern to enforce:**

```js
// 1. Standardise the escape helper — promote escHtml() to a shared utility
//    file (e.g. js/lib/pfc-escape.js) and import it everywhere instead of
//    redefining in each inline file. Currently defined in goals-2.js:15
//    AND debt-optimizer-2.js:252 (drift risk).

// 2. Add to .eslintrc.json:
{
  "plugins": ["no-unsanitized"],
  "rules": {
    "no-unsanitized/property": "error",
    "no-unsanitized/method":   "error"
  }
}
// This flags `el.innerHTML = anyVariable` and `el.innerHTML = \`...${var}...\``
// unless the value is a string literal or DOMPurify.sanitize() output.

// 3. Add a CI gate. In package.json:
{ "scripts": { "lint:security": "eslint 'js/**/*.js' --rule 'no-unsanitized/property:error'" } }
// Wire into Vercel build or a GitHub Action: fail PR if lint:security fails.
```

**Migration target for new code:** prefer `textContent` for any UI element that has no inner HTML structure. `el.textContent = userInput` is genuinely 100%-safe and has zero learning curve. Reserve innerHTML for code that must build sub-tree structure in one statement (rare in this codebase — most uses could be `textContent`).

### 3. Top-5 Code Changes (the actual fixes)

#### Fix #1 — `js/inline/goals-2.js:186` (the worst offender)

```diff
-        <div class="goal-info">
-          <div class="goal-name" title="${g.name}">${g.name}</div>
+        <div class="goal-info">
+          <div class="goal-name" title="${escHtml(g.name)}">${escHtml(g.name)}</div>
```

**Why this matters:** `g.name` is whatever the user typed into the "goal name" field on the goals page. Pre-Wave-11, an attacker with stored-XSS via a shared/imported goal could exfil `pfc_audit_session`. Post-Wave-11, they can inject `<style>` or fake-button HTML that looks like the app but links to a phishing domain. Still bad. Two-character fix.

#### Fix #2 — `js/inline/recurring-2.js:458`

```diff
-          <div class="rec-name">${r.name}</div>
-          <div class="rec-meta">Since ${firstFmt} · ${r.occurrences.length} charges · last ${lastFmt}</div>
+          <div class="rec-name">${escHtml(r.name)}</div>
+          <div class="rec-meta">Since ${escHtml(firstFmt)} · ${r.occurrences.length} charges · last ${escHtml(lastFmt)}</div>
```

Add `escHtml` definition to top of `recurring-2.js` (it doesn't have one). `r.name` derives from bank-transaction descriptions — those *can* contain `<` characters in real data (some merchants put HTML in their descriptors as a joke; payment processors don't sanitise; nobody expects you to render them as HTML). `firstFmt`/`lastFmt` are `toLocaleDateString` output and currently safe, but defensive — they ride through `escHtml` for negligible cost.

#### Fix #3 — Promote escHtml to a shared helper (kill drift)

**New file: `js/lib/pfc-escape.js`**

```js
// PFCEscape — single source of truth for HTML-escaping user-controlled strings
// before innerHTML interpolation. Use textContent when possible; this is for
// the cases where you need an HTML structure assembled in one template literal.
(function () {
  'use strict';
  window.PFCEscape = {
    html(s) {
      return String(s == null ? '' : s)
        .replace(/&/g, '&amp;')
        .replace(/</g, '&lt;')
        .replace(/>/g, '&gt;')
        .replace(/"/g, '&quot;')
        .replace(/'/g, '&#39;');
    },
    // For attribute values that may contain quotes — same logic, just named for clarity
    attr(s) { return this.html(s); }
  };
})();
```

Load this in every HTML page that uses `escHtml`. Then in inline JS, replace local `function escHtml(s) { ... }` definitions with `const escHtml = PFCEscape.html;`. Eliminates the goals-2.js / debt-optimizer-2.js drift.

#### Fix #4 — Sage chatbot reply path defense-in-depth

`sage-3.js:233-251` is currently safe (escape-then-markdown), but it's the highest-trust-boundary spot in the app. Add a regression test before you ship anything else:

```js
// Add to a sage.test.js or similar
const cases = [
  { input: 'hello <script>alert(1)</script>',                    must_not_contain: '<script>' },
  { input: 'click <a href="javascript:alert(1)">here</a>',       must_not_contain: 'javascript:' },
  { input: '<img src=x onerror=alert(1)>',                       must_not_contain: '<img' },
  { input: '**bold** with <b>raw</b>',                           must_contain:     '&lt;b&gt;' }
];
cases.forEach(c => {
  const out = _safeSageMarkdown(c.input);
  if (c.must_not_contain && out.includes(c.must_not_contain)) throw new Error('FAIL: '+c.input);
  if (c.must_contain && !out.includes(c.must_contain))         throw new Error('FAIL: '+c.input);
});
```

The order of replace operations matters: HTML-escape **first**, then markdown transform. If anyone ever reverses that order (e.g. "let's add link rendering"), this test catches it.

#### Fix #5 — `cash-forecast-2.js:419` (theoretical, do for hygiene)

```diff
-  sel.innerHTML = months.map(m => `<option value="${m.value}">${m.label}</option>`).join('');
+  sel.innerHTML = months.map(m => `<option value="${escHtml(m.value)}">${escHtml(m.label)}</option>`).join('');
```

Or better — rewrite to DOM API since it's a `<select>` populator:

```js
sel.replaceChildren(...months.map(m => {
  const opt = document.createElement('option');
  opt.value = m.value;
  opt.textContent = m.label;
  return opt;
}));
```

The second form is the canonical "do it properly" pattern and produces shorter, faster code. Recommended for any new code touching `<select>` / `<option>` / `<tr>` / `<td>`.

### 4. Trusted Types — `require-trusted-types-for 'script'`

**Verdict: Defer 6+ months. Useful eventually, not now.**

**What it does:** Tells the browser to refuse any string assignment to a "sink" (innerHTML, outerHTML, document.write, eval, setTimeout-with-string, etc.) unless the string is wrapped in a `TrustedHTML`/`TrustedScript` object minted by a registered policy. Defense-in-depth — catches innerHTML-injection bugs at runtime by *making them crash*, not by hoping the developer remembered to escape.

**What would break in this codebase if you enable it today:**

1. **Every innerHTML assignment that isn't a static string literal** — ~50–80 of the 117 sites would throw `TypeError: This document requires 'TrustedHTML' assignment`. Browser default policy doesn't exist; you'd need to register one.
2. **Inline templates everywhere** — every `innerHTML = \`<div>${x}</div>\`` becomes `innerHTML = trustedPolicy.createHTML(\`<div>${x}</div>\`)`. That's a 117-site refactor.
3. **Chart.js, Supabase SDK, html2canvas** — third-party libs that do their own innerHTML internally would break unless they're TT-aware. Chart.js 4.x is mostly safe. html2canvas does internal DOM mutation that has had TT issues historically.
4. **PayPal SDK** — `cdn.paypal.com/sdk/js?client-id=...` script does DOM-write internally; this would break payments. Verified: PayPal supports TT via a `data-csp-nonce` attribute mechanism but it's not zero-config.

**When to revisit:** Once you do the "migrate to ESM modules" refactor (which the audit also recommends), it's natural to add a single trusted policy at module-load time:

```js
const ttPolicy = window.trustedTypes?.createPolicy?.('pfc-default', {
  createHTML: (s) => s   // we trust our escHtml-wrapped strings
});
// Then everywhere: el.innerHTML = ttPolicy.createHTML(`<div>${escHtml(x)}</div>`);
```

Cost: ~4 hours to migrate, ~1 hour to debug whichever third-party lib breaks. Benefit: catastrophic-bug-prevention. Worth it post-PMF.

**Don't enable now.** It will eat half a day, all you get is "we already escaped everything anyway."

### 2. Hash Generation Command

For each pinned URL, fetch the file and compute the SHA-384 hash. Commands:

```bash
# Pattern (Bash, OpenSSL):
curl -s "https://cdn.jsdelivr.net/npm/@supabase/supabase-js@2.105.4/dist/umd/supabase.js" \
  | openssl dgst -sha384 -binary | openssl base64 -A
# Output: <hash>=
# Use as: integrity="sha384-<hash>=" crossorigin="anonymous"

# PowerShell (Windows, no openssl):
$bytes = (Invoke-WebRequest -Uri "https://cdn.jsdelivr.net/npm/@supabase/supabase-js@2.105.4" -UseBasicParsing).Content
$bytes = [System.Text.Encoding]::UTF8.GetBytes($bytes)
$sha = [System.Security.Cryptography.SHA384]::Create().ComputeHash($bytes)
[Convert]::ToBase64String($sha)
```

**Easier shortcut:** Use `https://www.srihash.org/` (paste URL, get tag). Or jsdelivr's own UI — `https://www.jsdelivr.com/package/npm/@supabase/supabase-js` has a "copy with SRI" button next to each version. Both faster than running openssl.

**Important:** Use `crossorigin="anonymous"` alongside `integrity` — without it the browser blocks the SRI check for cross-origin scripts.

### 4. Pinning vs Auto-Update Trade-off

| Choice | Security | Maintenance |
|--------|----------|-------------|
| Float (`@2`) | CDN compromise = you serve malware to users | Zero — bug fixes flow in automatically |
| Pin + SRI (`@2.105.4` + integrity) | CDN compromise = browser refuses to execute, page partially breaks but no malware delivered | You pick when to upgrade. Need a quarterly chore to bump versions |

**For a pre-PMF solo founder: pin everything.** The reasoning is asymmetric blast radius:
- Cost of auto-update breaking: Supabase ships a breaking change at 2am, your auth page silently dies, you discover it from a support email next morning. Real, painful, has happened to others.
- Cost of "I forgot to bump for 9 months": basically nothing for a stable lib. Supabase 2.105 → 2.118 is mostly bug fixes.
- Cost of CDN compromise with float: you ship malware to every user. Recovery = front-page news + every user has to assume their data is compromised.

**Pin + SRI is the right answer.** Add to your calendar: quarterly "bump CDN deps + recompute SRI" review (30 min, every 90 days).

## 6. Productive Paranoia — What You're NOT Working On

You asked: "With CSP locked, am I missing a more important threat?"

**Yes. Three things, ranked.**

### THREAT #1 (top priority) — Supabase RLS policy coverage

This is your real attack surface. CSP/SRI/innerHTML are *client-side* defenses. The server-side blast radius is "any authenticated user can read/write the wrong rows because RLS isn't enforced on table X." This is how most early-stage SaaS gets popped — not exotic XSS chains.

**Do this:**
- Run `select tablename, rowsecurity from pg_tables where schemaname='public';` against your Supabase DB.
- Every table with user data MUST have `rowsecurity = true`.
- Every table MUST have an explicit `using (auth.uid() = user_id)` policy or equivalent.
- Test it: create two test users, log in as user A, try to fetch user B's data with the JS client. If it returns rows, you have a critical bug.

ALE estimate (FAIR): probability 25% over 12 months (this is *the* common pre-PMF bug), impact $50K–$500K (data breach, GDPR notification, customer churn). Expected loss = ~$70K. SRI's ALE: probability < 0.1%, impact same. Expected loss = ~$300.

**RLS audit is ~200x more leveraged than SRI.** Do this first.

### THREAT #2 — Supabase service-role key exposure

If you have `SUPABASE_SERVICE_ROLE_KEY` in any Vercel env var that's bound to a client-facing function output (e.g. an Edge function that accidentally returns the key in an error response, or a debug endpoint), that's game-over. The service role bypasses RLS.

**Do this:**
- `git log --all -p | rg "SERVICE_ROLE|service_role"` — any leak in history?
- `rg "SERVICE_ROLE" api/` — does any endpoint reference it? If yes, audit each: does it ever return Supabase error messages to the client?
- The service-role key should ONLY be used in trusted server contexts that never echo Supabase errors verbatim.

### THREAT #3 — Account enumeration via auth endpoints

A `/api/auth/signup` that responds differently for "email taken" vs "email available" is a user-enumeration oracle. Supabase Auth has historically had subtle ones. Test: hit signup with a known-existing email and a known-fresh email; diff the responses (status code, timing, body, headers). If they differ, fix or rate-limit.

## Appendix A — Lint Rule Wiring

`.eslintrc.json` patch:

```json
{
  "plugins": ["no-unsanitized"],
  "rules": {
    "no-unsanitized/property": ["error", {
      "escape": { "methods": ["escHtml", "PFCEscape.html"] }
    }],
    "no-unsanitized/method": "error"
  }
}
```

Install: `npm i -D eslint-plugin-no-unsanitized`. The `escape.methods` array tells the plugin which wrapper functions to consider "safe" — so `el.innerHTML = \`<div>${escHtml(x)}</div>\`` passes lint while `el.innerHTML = \`<div>${x}</div>\`` fails.
