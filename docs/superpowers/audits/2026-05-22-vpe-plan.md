## ITEM A — Scoped E2E smoke tests (60 min)

### Why this, why now

303 handlers were converted via `scripts/convert-inline-handlers.js`. Mechanical regex passes have three failure modes the visual-regression harness CANNOT detect:

1. **Wrong arg order** — `data-pfc-arg-this data-pfc-arg='"emergency"'` ordering bug in bootstrap dispatcher means `toggleGoal(this, 'emergency')` vs `toggleGoal('emergency', this)`.
2. **Missing `this` context** — handlers like `togglePrivacy` rely on `this` being the button; if bootstrap dispatches with `this === window` they silently no-op.
3. **Listener registration timing** — if bootstrap runs before DOMContentLoaded for late-injected nodes (e.g. notification panel), clicks are dead.

Screenshots show none of these. A 2-click test does.

### Files to write

#### 1. `scripts/e2e-smoke.py` (NEW)

Decision: new file, not extension of `screenshot-photos.py`. Mixing screenshot-geometry-grep with click-flow assertions in one harness makes failure attribution muddy ("did clicking break, or did the figure detector break?"). Two small scripts beat one branchy script.

```python
"""
e2e-smoke.py — Two hand-picked click flows that MUST work post-Wave-11.

Flow A: Audit-mode entry
  /api/audit-login -> sets cookie -> /dashboard.html renders -> sample
  data visible (a known DOM token from the dashboard render path).

Flow B: Onboarding wizard step 1 -> step 6
  /onboarding.html -> click #btn-next five times -> #step-complete visible.

This proves that the data-pfc-on-* bootstrap dispatcher is wired
correctly for the two flows that gate the first-run experience. If
either flow breaks, the funnel is dead, regardless of how pretty
the screenshots look.

Run:  python scripts/e2e-smoke.py
Env:  AUDIT_BYPASS_TOKEN  (same as screenshot-photos.py)
Exit: 0 = both flows green; 1 = at least one flow red; 2 = config error.
"""
import os
import sys
from urllib.parse import quote as _urlquote
from playwright.sync_api import sync_playwright, TimeoutError as PWTimeout

BASE = os.environ.get("PFC_BASE_URL", "https://www.profinancecast.com")
TOKEN = os.environ.get("AUDIT_BYPASS_TOKEN")
if not TOKEN:
    print("FATAL: AUDIT_BYPASS_TOKEN not set", file=sys.stderr)
    sys.exit(2)

# Console error budget. Wave-11 cleanup should mean ZERO uncaught errors
# on these two flows. If you legitimately have a known-noisy 3rd party
# (cloudflare insights etc.) add a substring filter here, not a count bump.
ALLOWED_CONSOLE_ERRORS = []  # list of substrings; any error not matching => fail


def collect_console(page, sink):
    def on_msg(msg):
        if msg.type == "error":
            sink.append(msg.text)
    page.on("console", on_msg)
    page.on("pageerror", lambda e: sink.append(f"pageerror: {e}"))


def flow_a_audit_dashboard(context):
    """Audit login -> dashboard renders -> sample data visible."""
    errors = []
    page = context.new_page()
    collect_console(page, errors)

    # 1. Hit audit-login
    resp = page.goto(
        f"{BASE}/api/audit-login?t={_urlquote(TOKEN, safe='')}",
        wait_until="domcontentloaded",
    )
    if not resp or resp.status not in (200, 204, 302):
        return False, f"audit-login status={resp.status if resp else 'none'}", errors

    # 2. Dashboard renders
    page.goto(f"{BASE}/dashboard.html", wait_until="networkidle", timeout=20000)

    # 3. A known DOM token from the render path. #masthead-month gets
    # populated with current month abbreviation by dashboard render code;
    # if it's still "-" (em-dash placeholder), render did NOT run.
    try:
        page.wait_for_function(
            "() => { const el = document.getElementById('masthead-month'); "
            "return el && el.textContent.trim() && el.textContent.trim() !== '—'; }",
            timeout=8000,
        )
    except PWTimeout:
        masthead = page.evaluate(
            "() => document.getElementById('masthead-month')?.textContent ?? '(no element)'"
        )
        return False, f"dashboard did not hydrate; masthead-month={masthead!r}", errors

    # 4. Topbar date populated (proves the JS bundle ran end-to-end, not
    # just the masthead-month writer).
    today = page.evaluate("() => document.getElementById('today-date')?.textContent ?? ''")
    if not today.strip():
        return False, "today-date empty (JS bundle did not finish)", errors

    # 5. Console clean
    unexpected = [e for e in errors if not any(a in e for a in ALLOWED_CONSOLE_ERRORS)]
    if unexpected:
        return False, f"{len(unexpected)} console error(s); first: {unexpected[0]}", errors

    return True, "OK", errors


def flow_b_onboarding_wizard(context):
    """Onboarding: step-0 visible, click Next x5, step-complete visible."""
    errors = []
    page = context.new_page()
    collect_console(page, errors)

    page.goto(f"{BASE}/onboarding.html", wait_until="networkidle", timeout=20000)

    # Wave-11 ships the wizard with #step-0 already .active. Verify.
    active_initial = page.evaluate(
        "() => document.querySelector('.step-panel.active')?.id ?? null"
    )
    if active_initial != "step-0":
        return False, f"expected step-0 active on load, got {active_initial!r}", errors

    # Click Next five times: 0 -> 1 -> 2 -> 3 -> 4 -> 5
    expected = ["step-1", "step-2", "step-3", "step-4", "step-5"]
    for i, want in enumerate(expected, start=1):
        page.click("#btn-next")
        try:
            page.wait_for_function(
                f"() => document.querySelector('.step-panel.active')?.id === {want!r}",
                timeout=4000,
            )
        except PWTimeout:
            got = page.evaluate(
                "() => document.querySelector('.step-panel.active')?.id ?? null"
            )
            return False, f"click #{i}: expected {want}, got {got!r}", errors

    # One more click should advance to step-complete (the wizard's terminal panel).
    page.click("#btn-next")
    try:
        page.wait_for_function(
            "() => document.querySelector('.step-panel.active')?.id === 'step-complete'",
            timeout=4000,
        )
    except PWTimeout:
        got = page.evaluate(
            "() => document.querySelector('.step-panel.active')?.id ?? null"
        )
        return False, f"final click: expected step-complete, got {got!r}", errors

    unexpected = [e for e in errors if not any(a in e for a in ALLOWED_CONSOLE_ERRORS)]
    if unexpected:
        return False, f"{len(unexpected)} console error(s); first: {unexpected[0]}", errors

    return True, "OK", errors


def main():
    fails = 0
    with sync_playwright() as p:
        browser = p.chromium.launch(headless=True)
        context = browser.new_context(
            viewport={"width": 1440, "height": 900},
            device_scale_factor=1,
        )

        for name, fn in [("flow_a_audit_dashboard", flow_a_audit_dashboard),
                         ("flow_b_onboarding_wizard", flow_b_onboarding_wizard)]:
            ok, detail, errs = fn(context)
            status = "PASS" if ok else "FAIL"
            print(f"[{status}] {name}: {detail}")
            if errs:
                print(f"        console-errors-captured={len(errs)}")
                for e in errs[:3]:
                    print(f"          | {e[:200]}")
            if not ok:
                fails += 1

        browser.close()

    print(f"\n=== SUMMARY === fails={fails}")
    sys.exit(0 if fails == 0 else 1)


if __name__ == "__main__":
    main()
```

### Threshold defense

Numbers are picked to **lock the current baseline**, not chase scores. Pre-PMF, the cost of failing CI on a 3-point perf regression nobody can perceive >> the cost of waiting one more day to know about it.

| Metric | Threshold | Why this number |
|---|---|---|
| Performance | 0.85 (error) | Wave-4+8 baseline measured at ~92 on landing desktop. 0.85 = 7-point slack for genuine churn; below that = something regressed materially. |
| Accessibility | 0.90 (error) | A11y regressions are usually copy-paste of an unlabelled control. Easy to fix, cheap to catch. Hard threshold. |
| Best practices | 0.90 (warn) | Often dinged for 3rd-party cookies you do not control. Warn, do not block. |
| SEO | 0.90 (warn) | Pre-PMF SEO is not your bottleneck. Warn. |
| LCP | 2500ms (error) | Google's "good" threshold. Anything worse on a static site is a real regression. |
| CLS | 0.10 (error) | Layout shift is the single most-perceptible perf bug. Tight threshold justified. |
| FCP | 2000ms (warn) | Correlates with LCP; warn rather than double-block. |
| TBT | 300ms (warn) | Static site should be near 0; 300 leaves room for Sentry boot. |

### Mobile vs desktop

The config above is **desktop only** to keep the deferred ship cheap. Mobile preset adds run time (slower throttling) and triples flake risk on GitHub-hosted runners (CPU is shared, mobile throttling exaggerates noise). Add `--preset=mobile` as a second LHCI run only after the desktop one has been stable for 2 weeks.

### Vercel preview vs production decision

**Use production URL.** Hobby plan does not give you preview URLs without extra OIDC token plumbing, and the site is fully public + static — there is no "but it's behind auth on preview" gain. The 30-second post-deploy poll already exists.

## ITEM C — Sentry SLO alerts (60 min)

### SLO numbers and defense

For a pre-PMF static site with **~0 users**, "SLO" is partly fiction (you have no traffic to violate them) and partly a forcing function (the moment a real user shows up, alerts fire on the same thresholds you set today). Pick numbers that are:

- Conservative enough not to alarm-fatigue you with N=3 errors.
- Tight enough that when N=100 users hit, a real regression is caught.

| SLO | Threshold | Defense |
|---|---|---|
| Edge function success rate | 99.5% over 1h, alert on dip below | The Edge functions (`/api/audit-login`, anything Supabase-proxying) are the only server surface. 99.5% allows 1 in 200 to fail (transient Vercel cold-start or Supabase blip) without paging. Lower than that = real bug. |
| Client-side unhandled error rate | <2% of sessions over 1h | "2% of sessions hit an exception" is the inflection where DevTools-watching users start complaining. Above that == bad release. |
| Sentry event volume spike | >10x baseline in 5 min | At your scale, 10x is "something is auto-firing in a loop". Catches infinite-retry bugs faster than session-error percentages. |
| New issue first-seen on production | Any P0/P1 issue, alert immediately | While volume is low, every NEW issue is worth eyeballing within an hour. |
| Release health: crash-free sessions | >=99% for 24h post-deploy | Standard Sentry release-health metric. Below 99% = ship-stopper for next deploy. |

**What I'm explicitly NOT alerting on:**
- p95 latency. No traces (`tracesSampleRate: 0` per current `pfc-sentry.js`), so no signal.
- Apdex. Same reason.
- Specific endpoint error rates. Wait for traffic before you slice.

### Risk box

**If Sentry alerts fire false-positive (you wake up to noise):**
- **Impact:** alert fatigue, you start ignoring real ones. This is the #1 cause of observability dying.
- **Recovery:** the runbook has a "Tuning schedule" section with week-1 and week-4 review prompts. Tune the noisy rule OR add a specific `ignoreErrors` entry — never the lazy fix of "just disable the alert."
- **Worst case if you ignore tuning:** in 6 months Sentry costs $26/mo + you have a Pavlovian flinch at the email. Recoverable by spending one Saturday afternoon archiving rules.

**If Sentry alerts fail to fire on a real bug:**
- **Impact:** bug ships, user complains via email, you find out via human channel.
- **Recovery:** post-mortem question is "was the error captured at all?" If yes (in Sentry but no alert), the rule is too loose — tighten. If no (not in Sentry at all), the SDK is misconfigured — check `beforeSend` is not returning `null`, check the deny-urls regex, check the Edge function project has a DSN wired.

**If `autoSessionTracking: true` causes a measurable bundle/perf regression:**
- **Impact:** ~5KB extra bundle for session-pings. Negligible on a static site with the existing JS payload.
- **Recovery:** set to `false`, lose rule 5 (release health). Acceptable trade-off if you ever see it on the perf budget — which you will not.
